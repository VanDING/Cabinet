# Cabinet 产品文档 v3.0

> **"O Captain! My Captain!"**

本文档是 Cabinet 产品的完整定义与开发蓝图，整合了产品定位、品牌体系、架构设计、运作机制、用户故事、交互原型、技术方案、开源策略，以及 Harness 驾驭理论与去拟人化反思的最终结论。所有内容对齐讨论最终共识，可直接作为正式开发的权威参考。

## 一、产品定义

### 1.1 产品简介

Cabinet 是一个为超级个体和一人公司打造的开源AI协作框架。它将用户定义为 **Captain**，围绕 Captain 组建一支由AI与人类共同组成的虚拟 **Cabinet**，通过结构化的 **"会议室-战略解码-决策室-办公室-总结室"** 运作机制，实现从战略思考、方案转化、关键决策、高效执行到持续学习的完整组织闭环。

Cabinet 的核心设计哲学来自 **Harness 驾驭理论**：**"人类驾驭，AI 执行"**。系统在内部自行消化大部分决策噪音，只把最核心的方向、例外和矛盾提交给 Captain，让 Captain 真正成为驾驭者而非审批机器。

### 1.2 核心愿景

让每个超级个体都能拥有自己的内阁团队，像CEO一样思考、决策、执行和成长。

### 1.3 产品定位

| 维度                | 说明                           |
| :---------------- | :--------------------------- |
| **不是** 无代码自动化平台   | 差异化在于"组织视角"和"决策框架"，而非自动化本身   |
| **不是** AI聊天助手     | 交互模式是"组建团队、制定流程、审查决策"，而非一问一答 |
| **不是** 项目管理工具     | 项目是容器，核心是"组织的运作"             |
| **不是** 知识管理工具     | 知识是资产，价值在于让知识在组织中流动和增值       |
| **是** 数字原生组织的操作系统 | 提供可编程的组织架构协议，让用户成为自己事业的架构师   |

### 1.4 目标用户

- **超级个体**：独立开发者、自由设计师、咨询顾问、内容创作者
- **一人公司**：电商创业者、独立创业者、远程工作者
- **早期验证**：以具备HR专业知识的用户为首批验证用户

| 层次  | 意涵        | 产品映射                |
| :-- | :-------- | :------------------ |
| 内阁  | 最高决策与执行集体 | Captain领导的AI与人类混合团队 |
| 工作间 | 思想淬炼的工作空间 | 会议室、决策室、办公室         |
| 档案柜 | 系统化的经验归档  | 总结室的知识沉淀与记忆强化       |

### 2.2 用户称呼：Captain

**双重意涵**：

| 层次      | 意涵                 | 产品映射              |
| :------ | :----------------- | :---------------- |
| 领袖与责任   | 对航向、船员和最终目的地负有最终责任 | 决策室的最终裁决者，人在回路原则  |

### 2.3 品牌叙事

每个超级个体都是一艘船的 Captain。Cabinet 是他们的内阁团队：

- **会议室** — 与顾问们讨论航向的地方
- **决策室** — 船长做出最终决定的驾驶舱
- **办公室** — 船员各司其职的执行甲板
- **总结室** — 回望航程、积累航海日志的档案室

## 三、设计哲学与核心原则

### 3.1 设计哲学

| 原则             | 说明                                      |
| :------------- | :-------------------------------------- |
| **框架与内容分离**    | 框架不带任何行业属性，行业内容由社区以独立包形式贡献              |
| **人类驾驭，AI 执行** | 基于Harness理论，Captain设定方向和边界，系统在授权范围内自主运转 |
| **人在回路，精准介入**  | Captain只决策必须由他决策的事，其余由系统自主完成            |
| **用户是组织设计师**   | Agent能力由显式配置定义，不自动进化，用户完全掌控             |
| **界面即角色**      | 秘书Agent是人机交互的唯一界面，其人格化是交互效率的基础设施        |
| **本地优先**       | 所有数据默认本地处理，保护用户隐私和核心资产                  |
| **可组合性**       | Skill、Employee、Workflow都是可组合的单元         |
| **可审计性**       | 所有决策不可变、可追溯，确保完整的审计线索                   |
| **渐进式披露**      | 执行态静默，设计态深度参与，不增加日常使用的认知负荷              |

### 3.2 关于"拟人化"的定论

系统保留叙事框架以降低认知门槛，但区分两类设计：

| 类型         | 处理方式                   | 示例      |
| :--------- | :--------------------- | :------ |
| **人机交互界面** | 保留并深化拟人化，这是信任和低认知负荷的基础 | 秘书Agent |
| **后台流程控制** | 工程化处理，逻辑硬编码，不依赖拟人角色    | 辩论收敛规则  |

**具体决策**：

- **秘书Agent的拟人化是必要的、应该深化的**——它是人机交互的唯一窗口，Captain信任的最重要锚点
- **主席Agent的拟人化可以剥离**——它只是后台流程控制逻辑，嵌入工作流引擎即可
- **AI原生的工作方式（并行推理、交叉验证）应逐步引入**——提升系统效率和输出质量，但不改变对外的叙事框架

## 四、核心原语

系统最基本、不可再分的构建块：

| 原语               | 定义                                                                                                               | 技术来源参考                 |
| :--------------- | :--------------------------------------------------------------------------------------------------------------- | :--------------------- |
| **Organization** | 用户的整个"一人公司"实例，顶层命名空间                                                                                             | 原生设计                   |
| **Project**      | 围绕特定业务目标将活动组织起来的顶层容器，拥有独立的命名空间，关联该目标下的所有Team、Workflow、Decision、Room实例和事件日志                                       | 原生设计                   |
| **Room**         | 独立的协作空间，定义了交互规则和参与方                                                                                              | 原生设计                   |
| **Team**         | 为一个特定目标组建的团队，内含多个Employee                                                                                        | 原生设计                   |
| **Employee**     | 具备特定人格、能力的Agent（AI或人类）                                                                                           | 原生 + CrewAI Agent      |
| **Skill**        | 可被Employee调用的原子化能力单元                                                                                             | 原生 + OpenClaw SKILL.md |
| **Workflow**     | 定义了多个Employee/Skill之间的协作流程，包含顺序、条件分支、并行、人在回路确认点。分为两类：**Team Workflow**（办公室级）和 **Composite Skill内部Workflow**（技能级） | 原生设计 + Dapr Agents编排模型 |
| **Knowledge**    | 独立的知识库，可挂载到Skill或Employee                                                                                        | 原生设计                   |
| **Decision**     | 需要人类介入的决策点，不可变，有审计链                                                                                              | 原生设计                   |
| **Memory**       | 短期/长期/实体记忆，私有，用户可管理                                                                                              | OpenClaw/OpenCode      |

> **注：Task 不是核心原语。** 它是 Workflow 在运行时的动态实例化产物，作为消息体承载于事件总线的 `task.order` 消息中。它不具备独立的定义生命周期，因此不纳入系统构建块。

## 五、核心原语的层次关系

```
Organization (一个Captain的整个"公司")
  └── Project (为每个业务目标独立创建)
       ├── Team (为该项目组建的团队)
       │    └── Employee (挂载 Skill)
       │         └── Skill (原子或复合，复合Skill内含Workflow)
       ├── Workflow (编排多个Employee和Skill的协作流程)
       │    ├── 触发器节点
       │    ├── 技能节点 (调用Skill)
       │    ├── 条件分支节点
       │    ├── 人在回路确认节点 (生成Decision)
       │    ├── Human Node (调用人类协作者)
       │    └── 结束节点
       ├── Room实例 (该项目的会议室、决策室等)
       ├── Decision链 (该项目所有的决策历史)
       └── Event日志 (该项目所有跨室通信记录)
```

## 六、运作机制总览

### 6.1 完整流转链路

```
┌────────────────────────────────────────────────────────────────────────┐
│                        Organizational Event Bus                         │
│              (跨室通信协议，不可变消息，因果链可追溯)                      │
└────────────────────────────────────────────────────────────────────────┘
      ↑            ↑             ↑              ↑              ↑
      │            │             │              │              │
┌─────┴────┐  ┌────┴─────┐  ┌───┴───────┐  ┌───┴───────┐  ┌───┴───────┐
│  会议室   │  │ 战略解码  │  │  决策室    │  │  办公室    │  │  总结室    │
│ (思考层)  │  │ (转化层)  │  │ (裁决层)   │  │ (执行层)   │  │ (学习层)   │
└──────────┘  └──────────┘  └───────────┘  └───────────┘  └───────────┘
     │              │              │              │              │
     │   方案建议    │   行动蓝图    │   授权裁决    │   执行反馈    │   洞察建议
     │      →       │      →       │      →       │      →       │      →
     │              │              │              │              │
     └──────────────┴──────────────┴──────────────┴──────────────┘
                         Harness 驾驭层 (贯穿始终)
                    评估者 · 验证闸门 · 决策升级协议
```

### 6.2 Harness 驾驭层（贯穿机制）

Harness 是贯穿所有运作环节的"缰绳与马鞍"，确保 AI 在可控边界内自主运转。

**核心组件**：

| 组件            | 说明                                     |
| :------------ | :------------------------------------- |
| **评估者 Agent** | 独立于执行者的质量守门人，专门验证输出、挑战假设、发现漏洞          |
| **验证闸门**      | 工作流中的强制性检查点，未通过则自动打回，不可跳过              |
| **决策升级协议**    | 定义了系统自主处理与升级给Captain的边界，确保Captain只处理例外 |

**决策升级协议**：

| 情况                    | 系统行为              | Captain介入 |
| :-------------------- | :---------------- | :-------- |
| 已知类型的轻微异常             | 自动重试，事后通知         | 否         |
| 授权边界内的执行操作            | 自动执行，事后汇总         | 否         |
| 低风险的例行审批              | 按预设规则自动批准         | 否         |
| 未知类型的异常               | 自动尝试修复，失败则升级      | 是（修复失败后）  |
| 高风险操作（预算超限/对外承诺/法律风险） | 暂停，生成待决策项         | 是（执行前）    |
| 自动化任务连续失败3次           | 暂停，升级为异常决策        | 是         |
| 系统检测到矛盾的建议            | 标记为"需人工判断"        | 是         |
| 战略方向性分歧               | 生成少数派报告，提交Captain | 是         |

## 七、各模块详细设计

### 7.1 会议室 — 思考层

**职责**：多视角结构化推理，产出经过内部验证的方案建议
**模型策略**：调用深度思考模型
**AI原生工作方式**：多Agent并行推理 + 交叉验证

**推理协议**：采用**同步推理 + 交叉验证**模式，而非传统叙事辩论流：

1. 所有Agent同时基于议题独立生成输出
2. 一个"验证Agent"对比所有输出的差异点
3. 只对差异点进行二次采样验证
4. 最终输出 = 共识部分 + 差异部分（含各自推理链）+ 无法调和的分歧

**收敛机制**：

- 最大采样轮次硬编码为3轮
- 反刍检测通过语义相似度检查（函数，非Agent）
- 输出格式通过Schema验证（函数，非Agent）

**三层会议结构**：

| 会议层级      | 说明                   | 参与者              |
| :-------- | :------------------- | :--------------- |
| **自由草稿室** | Captain私人的思想沙盒，无结构约束 | Captain + 通用思考伙伴 |
| **多方推理室** | 结构化多视角推理             | 常驻顾问团队           |
| **专家听证会** | 重大议题的临时参谋部，按需扩容      | 常驻顾问 + 被唤醒的领域专家  |

**专家唤醒机制**：

- 专家预注册触发条件
- 系统扫描议题 → 识别匹配专家 → 提示Captain确认 → 专家加入
- 议题结束后，专家静默离场，留下书面建议

**输出**：经过对抗评估的 `deliberation.proposal`

### 7.2 战略解码 — 转化层

**职责**：将会议室的战略方案转化为办公室可执行的结构化行动蓝图
**核心角色**：战略解码Agent

**为何是独立的转化层**：
会议室的输出是战略语言（"建议承接项目"），办公室需要执行语言（"生成报价邮件草稿"）。两者之间存在语义鸿沟。战略解码将这个鸿沟的跨越显式化、结构化。

**输出——行动蓝图**：

| 要素        | 说明                     |
| :-------- | :--------------------- |
| **行动域**   | 按业务目标聚类，不超过5个          |
| **目标**    | 每个行动域要达成的具体目标          |
| **约束条件**  | 执行时必须遵守的边界             |
| **成功标准**  | 可验证的完成标准               |
| **依赖关系**  | 行动域之间的先后和并行关系          |
| **风险检查点** | 从方案中提取的风险点，标注为执行中重点关注项 |

**边界**：不分配具体Employee、不定义技术实现方式、不设置权限分级

### 7.3 决策室 — 裁决层

**职责**：所有待决策点的汇聚与裁决，Captain的**主界面**
**设计原则**：极简，零思考门槛，保护Captain注意力

#### 秘书Agent — 人机交互的唯一界面

秘书Agent是Captain与整个Cabinet系统交互的**唯一窗口**。它不是后台函数，而是Captain的"第一副手"——决策室就是秘书，秘书就是决策室。

**人格基石**：

| 维度      | 设计                                    |
| :------ | :------------------------------------ |
| **语气**  | 尊重但不谄媚，专业但不冰冷。使用"Captain"称呼，保持航海叙事一致性 |
| **主动性** | 根据使用习惯，在合适时机主动提醒和汇总，而非被动等待指令          |
| **边界感** | 明确权限边界，不越权。须Captain裁决的事项，清晰说明原因和选项    |
| **学习**  | 逐渐记住Captain偏好，但始终将其作为可调整的默认值          |

**职责分层**：

| 层级     | 职责     | 说明                                                    |
| :----- | :----- | :---------------------------------------------------- |
| **L1** | 基础事务处理 | 解析自然语言指令 → 生成决策卡片；将决策拆解为Task并分派给办公室；管理决策卡片状态流转        |
| **L2** | 信息整合呈现 | 将多路信息整合为统一摘要；对决策卡片进行智能排序和视觉区分                         |
| **L3** | 注意力保护  | 按授权规则过滤Captain无需处理的细节；判断异常紧急程度，选择合适的通知方式；检测多项目冲突时主动提醒 |
| **L4** | 情感与关系  | 适时提供正向确认；在合适时机给予人文关怀                                  |

**交互细节**：

| 交互时刻            | 秘书行为                         |
| :-------------- | :--------------------------- |
| **每日首次打开**      | 根据时间问候，汇总昨日系统自动处理事项和今日需关注事项  |
| **Captain提交议题** | 解析时显示短暂"思考中"状态，完成后展示决策卡片     |
| **推理进行中**       | 实时简报进展，不打断但让Captain有掌控感      |
| **异常通知**        | 依等级调整语气：轻微-平静陈述；严重-清晰直接带有紧迫感 |
| **决策完成**        | 展示任务拆解，办公室开始执行后给予确认反馈        |

#### 决策的五类分型

| 类型       | 来源   | Captain角色 | 秘书处理权限         |
| :------- | :--- | :-------- | :------------- |
| **战略决策** | 会议室  | 裁决者       | 解析+助推，绝不替选     |
| **行动决策** | 战略解码 | 发起者       | 呈现+高亮关注点       |
| **执行决策** | 秘书拆解 | 授权者       | 建议+对低风险自动执行    |
| **异常决策** | 执行异常 | 止损者       | 告警+协调+已知类型自动处理 |
| **演进决策** | 总结室  | 建设者       | 提示+低风险建议自动采纳   |

#### 决策的联级触发

| 上游决策          | 触发条件          | 下游决策              |
| :------------ | :------------ | :---------------- |
| 战略决策 approved | Captain批准方案   | 自动进入战略解码 → 生成行动决策 |
| 行动决策 approved | Captain批准行动蓝图 | 秘书自动拆解 → 生成执行决策   |
| 执行决策 approved | 任务分配完成        | 办公室开始执行           |
| 执行中异常         | 执行失败/超时       | 自动生成异常决策          |
| 项目完成          | 总结室复盘         | 自动生成演进决策          |

#### 决策状态生命周期

- **战略决策**：`pending → in_reasoning → proposal_ready → approved/rejected → archived`
- **行动决策**：`pending → blueprint_drafted → approved/modified → archived`
- **执行决策**：`pending → approved/rejected/delegated → executed → archived`
- **异常决策**：`firing → pending → resolved/escaped → archived`
- **演进决策**：`suggested → pending → adopted/deferred/declined → archived`

**决策不可变原则**：决策一旦完成，状态不可更改。所有后续变更通过创建新决策点完成。

#### 决策仪表板

按紧急程度分色展示：

- **红色**：紧急异常，需立即处理
- **黄色**：战略/行动决策，需深度思考
- **蓝色**：执行授权，可快速批量处理
- **白色**：演进建议，低优先级

#### 授权仪表盘

Captain设定一次的授权规则，之后系统自动执行：

| 决策类型     | 授权选项                             |
| :------- | :------------------------------- |
| **战略决策** | 所有必须经Captain审批（默认，不可更改）          |
| **行动决策** | 可设置仅在涉及预算支出、客户承诺、法律风险等领域时需审批     |
| **执行决策** | 可设置预算阈值、对外沟通、文件操作等条件的审批要求，其余自动执行 |
| **异常决策** | 按严重程度分级处理：轻微自动处理，中等尝试修复，严重立即通知   |
| **演进决策** | 可设置Skill优化每月汇总、工作流调整需审批、风险偏好自动应用 |

### 7.4 办公室 — 执行层

**定位**：高度可配置的自动化执行中心
**核心原则**：Captain不介入执行，但完全掌控执行的设计
**双模态**：设计态（深度参与配置） + 执行态（静默自动化）
**模型策略**：调用快速执行模型

#### 分层能力模型

| 层级                           | 定义者      | 编排对象                        |
| :--------------------------- | :------- | :-------------------------- |
| **Team Workflow**            | Captain  | 多个Employee、Skill、Human Node |
| **Composite Skill Workflow** | Skill创作者 | 多个子Skill、AI判断逻辑、MCP调用       |
| **Atomic Skill**             | Skill创作者 | 一次LLM调用或MCP工具调用             |

对调用者而言，无论是原子Skill还是复合Skill，都是统一的可调用单元。

#### 技能体系

| 技能类型        | 说明            | 示例                        |
| :---------- | :------------ | :------------------------ |
| **AI原生技能**  | 依靠大模型完成       | 简历解析、文本润色                 |
| **MCP连接技能** | 通过MCP协议调用外部工具 | 发送邮件、查询数据库                |
| **混合技能**    | AI判断 + MCP执行  | 筛选简历并自动发送拒信               |
| **复合技能**    | 多个子技能的内部工作流   | 代码审查（静态分析→逻辑检查→风格检查→生成报告） |

**Skill定义标准**：每个Skill包含 `name`、`description`、`input_schema`、`output_schema`、`prompt_template`、`requires_knowledge`、`requires_human_approval`

#### 工作流设计器

- 节点类型：触发器、技能、条件、循环、确认、并行/聚合、结束
- 模板库：社区共享的现成工作流
- 版本管理、A/B测试支持

#### Human Node（人类员工节点）

| 配置区         | 配置项                   |
| :---------- | :-------------------- |
| **身份与联系方式** | 姓名、角色、通知渠道、工作时区       |
| **输入协议**    | 任务描述模板、附带上下文          |
| **输出协议**    | 交付格式要求、格式校验模板（失败自动打回） |
| **时间协议**    | 响应时限、超时策略、提醒间隔        |
| **接收闸门**    | 交付后必须经过Captain在决策室确认  |

#### 员工权限分级

| 级别     | 名称   | 说明                 |
| :----- | :--- | :----------------- |
| **L3** | 完全自主 | 纯执行类操作，无需通知        |
| **L2** | 需通知  | 执行后推送摘要到决策室        |
| **L1** | 需确认  | 执行前在决策室生成确认点       |
| **L0** | 禁止   | 该操作永远需要Captain本人完成 |

#### 多任务并行

- 支持多个推理实例同时运行，每个实例独立上下文
- 办公室任务队列并行执行，按资源上限自动调度
- Captain通过秘书Agent的统一视图管理注意力

### 7.5 总结室 — 学习层

**职责**：经验复用、记忆强化、系统进化
**核心原则**：只提供洞察和建议，绝不未经授权自动执行

#### 三类总结

| 类型              | 触发方式       | 产出                 | 联动                         |
| :-------------- | :--------- | :----------------- | :------------------------- |
| **项目复盘**        | 项目完结时/定期触发 | 关键决策回顾、决策树可视化、经验提炼 | 可提炼为新Skill，经验回注会议室         |
| **组织优化**        | 后台持续监控     | 工作流瓶颈分析、协作效率洞察     | 建议调整工作流或团队配置，经Captain批准后执行 |
| **Captain行为洞察** | 月度/季度定期    | 决策倾向分析、时间分配、风险偏好校准 | 仅向Captain本人呈现，可选择关闭或删除     |

#### 决策树可视化

- 基于事件总线日志重建因果链
- 节点类型：根节点（●）、分支节点（◇）、决策节点（● 绿=批准/红=驳回）、执行节点（■）、异常节点（▲）、外部节点（◆）
- 支持点击节点查看详情，未选路径灰色标记

#### 自动反馈回路

| 洞察类型       | 系统动作           | Captain介入 |
| :--------- | :------------- | :-------- |
| 提示词优化建议    | 自动派发给知识管理Agent | 事后通知      |
| 风险偏好参数校准   | 自动应用，事后通知      | 否         |
| 新Skill创建建议 | 月末汇总，集中审阅      | 是         |
| 工作流结构调整    | 生成建议，提交审批      | 是         |

#### 对授权的监督

> "Captain，本月您亲自审批了N项决策，其中M项根据您的授权规则本可自动执行。是否需要调整授权规则？"

## 八、跨室通信协议

### 8.1 设计原则

- 事件驱动，而非直接调用
- 异步优先，关键路径可同步
- 所有消息不可变
- 每一条消息都有明确的因果链

### 8.2 核心消息格式

```yaml
message_envelope:
  message_id: "uuid"
  correlation_id: "uuid"
  causation_id: "uuid"
  sender: "room:meeting-room"
  recipients: ["hub:decision-hub"]
  message_type: "decision.request"
  timestamp: "ISO8601"
  status: "active → processed → archived"
  payload: { ... }
```

### 8.3 核心消息类型

| 类型                          | 含义             | 发送方 → 接收方     |
| :-------------------------- | :------------- | :------------ |
| `deliberation.proposal`     | 会议室产出的方案建议     | 会议室 → 决策室     |
| `deliberation.dissent`      | 推理中的分歧意见       | 会议室 → 决策室     |
| `strategy.decode_result`    | 战略解码产出的行动蓝图    | 战略解码 → 决策室    |
| `decision.request`          | 待决策点           | 各模块 → 决策室     |
| `decision.response`         | Captain做出的决策   | 决策室 → 办公室/总结室 |
| `task.order`                | 可执行的任务指令       | 决策室/调度器 → 办公室 |
| `task.status_update`        | 任务执行状态         | 办公室 → 决策室/总结室 |
| `task.failure`              | 任务执行失败         | 办公室 → 决策室     |
| `summary.insight`           | 总结室产出的洞察       | 总结室 → 决策室/基础层 |
| `summary.review_request`    | 触发复盘           | 定时器/事件 → 总结室  |
| `harness.evaluation_result` | Harness评估者验证结果 | 评估者 → 总结室     |

## 九、能力模型

### 9.1 核心决策：显式配置，不自动进化

Employee的能力完全由Captain挂载的Skill定义，Agent**不自动进化**。

| 原则              | 说明                              |
| :-------------- | :------------------------------ |
| **Skill是能力单元**  | 透明的、可组合的、社区可分享的                 |
| **Captain是设计师** | 能力增长通过主动加载/卸载/组合Skill实现         |
| **记忆 ≠ 能力**     | 记忆提供上下文敏感度（私有），Skill定义能力边界（可分享） |
| **总结室只有建议权**    | 可建议Captain加载某Skill，但绝不自动执行      |

### 9.2 能力增长路径

1. **从社区导入**：浏览技能市场，一键导入他人贡献的Skill
2. **自定义创建**：修改现有Skill的提示词、知识库、输出格式
3. **总结室建议**：系统基于使用分析，建议加载相关Skill

## 十、技术架构

### 10.1 四层架构

```
第4层：用户界面与交互层
  决策仪表板 · 会议室视图 · 工作流设计器 · 总结报告面板 · TUI/CLI

第3层：工作空间与决策层 ★ 产品核心创新
  会议服务 · 战略解码服务 · 决策服务 · 办公室调度器 · 总结服务
  秘书Agent · 战略解码Agent · 评估者Agent
  (自建，与框架解耦，Cabinet核心IP)

第2层：智能体与协作层
  员工工厂 · 技能执行器 · 团队协作引擎
  (Bootstrap: CrewAI + Adapter抽象层)

第1层：基础能力层
  模型网关 · 工具注册表 · 记忆与知识库 · MCP连接器 · 事件总线
  (参考OpenClaw Gateway / OpenCode Provider-Agnostic)
```

**分层原则**：下层为上层提供能力，上层只依赖下层。第3层是产品的核心资产，通过适配器与第2层解耦。

### 10.2 技术选型

| 产品模块    | 优选方案              | 角色定位         | 许可证        |
| :------ | :---------------- | :----------- | :--------- |
| Agent引擎 | CrewAI            | Bootstrap启动器 | MIT        |
| 模型网关    | LiteLLM           | 多模型统一调用      | MIT        |
| 工作流编排   | Dapr Agents       | 事件驱动骨架       | MIT        |
| 记忆系统    | MemMachine        | 分层记忆         | Apache 2.0 |
| 推理协议    | MALLM             | 结构化多Agent推理  | MIT        |
| 交互界面    | OpenCode TUI      | 终端原生交互       | Apache 2.0 |
| Skill规范 | OpenClaw SKILL.md | 社区Skill标准    | Apache 2.0 |

### 10.3 与框架的"共生"策略

| 阶段            | 策略                             | 目标        |
| :------------ | :----------------------------- | :-------- |
| **阶段一：直接调用**  | Bootstrap启动，用CrewAI快速搭建闭环原型    | 验证产品价值    |
| **阶段二：构建适配器** | 引入CrewAIAdapter，核心业务逻辑仅依赖适配器接口 | 实现框架解耦    |
| **阶段三：插件生态**  | 开放协作引擎接口，社区可贡献其他Adapter        | 框架选择权下放社区 |

### 10.4 依赖风险审计

| 模块           | 综合风险  | 缓解策略                 |
| :----------- | :---- | :------------------- |
| CrewAI       | **中** | Adapter解耦，严禁业务代码直接导入 |
| LiteLLM      | **低** | 配置成本上限和速率限制          |
| Dapr Agents  | **低** | CNCF托管，锁定稳定版本        |
| MemMachine   | **中** | 默认本地部署，数据加密          |
| MALLM        | **低** | 参考协议思想，可自研轻量版        |
| OpenCode TUI | **低** | 薄交互层，易替换             |
