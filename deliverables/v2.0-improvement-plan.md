## 二、阶段 B: 控制论闭环补强 (2 周)

### B.1 Blackboard Mid-Session 推送 (原则 4, P0)

**问题**: Blackboard snapshot 仅在 `buildContext` 时注入 system prompt。如果 Agent B 在 Agent A 写入 discovery 之前已启动，Agent B 在整个 session 中看不到 A 的更新。

**方案**:

```
Agent A write discovery
  → BlackboardTopicRouter.publishTopic()
  → EventBus 广播 SystemNotification
  → 活跃 AgentLoop 的 HandoffObserver 检测 zone change
  → 在下一个 step 的 context 中追加 [Shared Context Update]
```

**实施**:

| 任务                                                                              | 工时 | 验收                  |
| --------------------------------------------------------------------------------- | :--: | --------------------- |
| BlackboardObserver (新) — 订阅 Blackboard 事件, 写入 ctx.pendingBlackboardUpdates |  4h  | 单元测试              |
| ContextBuilder 修改 — 每步 context assembly 时 check pending updates              |  3h  | 增量注入 < 500 tokens |
| 端到端测试 — 两个 AgentLoop 实例, A 写入后 B 在 3 步内感知                        |  4h  | 延迟 < 3 steps        |

**合计: 11h**

### B.2 PIS Intent Alignment 升级到 Embedding (原则 2, P1)

**问题**: Phase 1 的 Jaccard keyword overlap 对语义相近但用词不同的任务会误判 drift。

**方案**: 注入 `EmbeddingService` (已在 `@cabinet/gateway` 可用)。保留 keyword Jaccard 作为 fallback。

**实施**:

| 任务                                               | 工时 | 验收                                 |
| -------------------------------------------------- | :--: | ------------------------------------ |
| `calculateIntentAlignment` 新增 embedding 分支     |  4h  | embedding 可用时 cosine > 0.8 为高分 |
| 降级逻辑 — embedding 不可用时 fallback 到 keyword  |  2h  | 0 embedding cost 模式 via config     |
| 更新 PIS 测试 — 语义相近但用词不同的 case 不应误判 |  4h  | 新测试通过                           |

**合计: 10h**

### B.3 AdaptiveContextMonitor 多维度阈值 (原则 2+7, P1)

**问题**: 当前只用 utilization 一个维度，同角色不同任务的最优阈值不同。

**方案**: 扩展为 `(model, role, taskCategory)` 三维。taskCategory 从 task 描述的前 200 字符做简单分类（code / analysis / conversation / search）。

**实施**:

| 任务                                                          | 工时 | 验收           |
| ------------------------------------------------------------- | :--: | -------------- |
| `getZonePerformance` 扩展 — 加入 taskCategory filter          |  3h  | query 支持三维 |
| `recalibrate` 扩展 — per-(model, role, category) threshold    |  4h  | 三维阈值表     |
| 样本不足时降级 — category 不足 → role 级 → model 级 → default |  3h  | 降级链测试     |

**合计: 10h**

### B.4 Tool Variety 定量度量 (原则 7, P1)

**问题**: 系统不知道当前暴露多少 tool、实际使用多少、gap 多大。

**方案**: 新增 `ToolVarietyMetrics` — 每次 session 结束时记录。

**实施**:

| 任务                                                                        | 工时 | 验收                 |
| --------------------------------------------------------------------------- | :--: | -------------------- |
| `ToolVarietyCollector` — 收集 exposedTools / usedTools / uniqueToolsPerStep |  3h  | session summary 包含 |
| Dashboard 展示 — RuntimeDashboard 新增 "Tool Variety" 卡片                  |  3h  | 可视化               |
| 告警 — gap > 5x 时 SystemNotification                                       |  2h  | 事件发布             |

**合计: 8h**

**阶段 B 合计: 39h (~1 周)**

## 八、验收标准 (v2.0 整体)

1. **[表征测试]** AgentLoop + Dispatcher 表征测试全部通过, 与 golden file 差异 < 5%
2. **[Blackboard]** 两个 AgentLoop 实例, A 写入 discovery 后 B 在 ≤3 steps 内感知
3. **[PIS]** embedding 模式下, 语义相近但用词不同的 task 不被误判为 drift (< 10% 误报)
4. **[Adaptive]** per-(model, role, taskCategory) 三维阈值表生成, 降级链完整
5. **[Tool Variety]** Dashboard 展示 exposedTools / usedTools / gap ratio
6. **[削冗]** 6 个未使用依赖移除, GC/BrowserPool 代码移除, meeting/organize 目录清理
7. **[文档]** 过时文档删除/更新, Blackboard + PIS 新文档, CABINET.md 同步
8. **[前端]** RuntimeDashboard 展示 PIS + Blackboard 状态, Settings 可配置 Adaptive + PIS
9. **[基线]** typecheck 零错误, 全量测试通过, 核心性能指标不劣化于 v0.9.0 基线
